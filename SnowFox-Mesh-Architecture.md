Systemübersicht & Design-PhilosophieSnowFox-Mesh ist ein dezentrales, hochgradig ausfallsicheres Peer-to-Peer-Kommunikationsnetzwerk, das als integriertes Feature innerhalb von SnowFoxOS (einem spezialisierten Linux-System) betrieben wird.Im Gegensatz zu staatlichen Krisensystemen oder rein hobbybasierten Funknetzwerken setzt SnowFox-Mesh auf eine Zwei-Klassen-Gerätearchitektur:Die Infrastruktur-Knoten (SnowFoxOS-Geräte): Leistungsfähige Linux-Knoten (z. B. auf Basis alter Android-Smartphones, Raspberry Pis oder Mini-PCs), welche die Brücken zwischen den Endgeräten und die Weitstrecken-Funkverbindungen herstellen.Die Endnutzer (Unmodifizierte Client-Geräte): Jedes herkömmliche Smartphone oder Laptop, das ohne App-Installation oder Hardware-Zubehör sofort über einen Standard-WLAN-Browser am Netzwerk teilnehmen kann.2. Weg 1: Standard-WLAN (Client-Access-Point)Technologie: $802.11\text{b/g/n}$ Access Point (gehostet direkt über die On-Board-WLAN-Karte des SnowFoxOS-Geräts).Funktion: Spannt eine lokale $50\text{-Meter}$-Blase auf. Smartphones und Laptops der Bevölkerung verbinden sich hiermit. Über ein integriertes Captive Portal (DNS-Redirection) wird jede HTTP-Anfrage abgefangen und das lokale, responsive Web-Interface von SnowFoxOS im Browser des Nutzers geöffnet.Weg 2: LoRa auf $2.4\text{ GHz}$ (Das unbeschränkte Nahbereichs-Mesh)Technologie: Spezielle Funk-Transceiver (z. B. Semtech SX1280), betrieben als externes USB-Modem mit RNode-Firmware.Funktion: Da im globalen $2.4\text{-GHz}$-Band keine Duty-Cycle-Beschränkungen (Sendelimits) existieren, nutzt SnowFoxOS diesen Weg für den kontinuierlichen Datenaustausch, lokale Gruppen-Chats und regelmäßige Synchronisationen zwischen benachbarten Knoten im Umkreis von bis zu $1.5\text{ Kilometern}$.Weg 3: LoRa auf $868\text{ MHz}$ (Das Weitstrecken-Rückgrat)Technologie: Sub-GHz-Transceiver (z. B. Semtech SX1262 / Heltec V3 / T-Beam).Funktion: Bietet die maximale physikalische Reichweite (durchdringt dicke Betonwände und bewaldetes Gebiet mühelos über Kilometer). Aufgrund der gesetzlichen $1\%$-Duty-Cycle-Regelung (max. $36\text{ Sekunden}$ Sendezeit pro Stunde) wird dieser Kanal vom Go-Core streng überwacht und ausschließlich für hochpriorisierte Alarme, behördliche Anweisungen und das grundlegende RNS-Routing verwendet.3. Protokoll-Schicht: Reticulum-Integration (RNS)SnowFox-Mesh verzichtet im dezentralen Backhaul (Weg 2 & 3) vollständig auf das traditionelle TCP/IP-Modell. Stattdessen bildet Reticulum (RNS) die logische Transportschicht.Die "Zero-DHCP"-ArchitekturIm Katastrophenfall bricht traditionelles IP-Routing zusammen, da IP-Zuweisungen (DHCP) und DNS-Server fehlen. SnowFoxOS löst dies elegant:IPv6 Link-Local: Das WLAN-Interface (Weg 1) vergibt keine klassischen IP-Adressen. Stattdessen weisen sich die verbindenden Handys automatisch selbst eine Link-Local IPv6-Adresse (fe80::/10) zu.Reticulum AutoInterface: RNS und der Go-Kern binden sich direkt an dieses IPv6-Link-Local-Interface. RNS benötigt keine IP-Adressen oder Gateways, um Routen zu berechnen. Es nutzt kryptografische Hashes als Adressen. Jedes verbundene Handy ist sofort im RNS-Netzwerk sichtbar – ohne, dass eine einzige IP konfiguriert werden musste.Paket-Tarnung & Krypto-IsolationKeine Beacons: Im Gegensatz zu WLAN oder Mobilfunk sendet Reticulum keine dauerhaften Such-Signale ("Beacons"). Das Funknetzwerk verhält sich vollkommen stumm. Erst beim tatsächlichen Absenden einer Nachricht blitzt das Signal für wenige Millisekunden auf.Unsichtbare Pakete: Da Reticulum den Datenstrom über das Noise Protocol Framework verschlüsselt, sind alle Paket-Header, Routen-Informationen und Payloads für feindliche Spektrumanalysatoren ununterscheidbar von normalem, weißem Rauschen. Ein Angreifer kann nicht feststellen, ob das Signal ein Rauschen der Atmosphäre, eine Mikrowelle oder eine hochpriorisierte SnowFox-Nachricht ist.4. Das 5-Prioritäten-System & Battery-AwarenessDa das Netz im Ernstfall von schnellem WLAN auf extrem langsame LoRa-Kanäle umschalten muss, filtert und steuert der Go-basierte Hintergrunddienst von SnowFoxOS den Datenverkehr über eine Priority Queue (Prioritätswarteschlange).[HÖCHSTE PRIORITÄT]
  ▲  P5: Lebenswichtige Notrufe / Behörden-Sirenensignale (Bypass-Warteschlange)
  │  P4: RNS-Infrastruktur / Routing-Tabellen (Pfad-Updates)
  │  P3: Direkte P2P-Nachrichten (Verschlüsselt, Familie)
  │  P2: Öffentliche Stadtteil-Chats (Gedrosselt / Verzögert)
  ▼  P1: Telemetrie / Bilder / Profildateien (Im Katastrophen-Modus gesperrt)
[NIEDRIGSTE PRIORITÄT]
Der "Geiz-Modus" (Battery Awareness)Die Knoten messen kontinuierlich ihre verbleibende Energieversorgung (z. B. den Akkustand des gehosteten Alt-Smartphones oder die Kapazität der Pufferbatterie). Fällt der Akku unter $20\%$, schaltet der Knoten in den ressourcenschonenden Schutzmodus:// Schematischer Go-Code für die Battery-Awareness-Entscheidung
func ShouldForwardPacket(priority int, batteryPercentage int) bool {
    if batteryPercentage < 20 {
        // Unter 20% Akku werden nur lebenswichtige Notrufe und Routing-Keep-Alives weitergeleitet
        return priority >= 4
    }
    if batteryPercentage < 50 {
        // Unter 50% Akku werden keine ressourcenintensiven Chat-Pakete mehr weitergeleitet
        return priority >= 3
    }
    return true // Normalbetrieb
}
Effekt: Die Lebensdauer eines sterbenden Knotens verlängert sich von wenigen Stunden auf mehrere Tage, da er nur noch bei P5-Signalen (z. B. einem aktiven SOS-Ruf in der Nachbarschaft) für Millisekunden den Sender aktiviert.5. 1. DNS-Spoofing & Captive PortalDer Go-basierte DNS-Dienst fängt alle Standard-Verbindungstests von Smartphones (wie connectivitycheck.gstatic.com für Android oder captive.apple.com für iOS) ab. Der Server antwortet auf jede Anfrage mit der IP des lokalen SnowFoxOS-Webservers (10.42.0.1). Das Smartphone glaubt, in einem Hotel-WLAN zu sein, und öffnet automatisch das Browser-Fenster.2. Progressive Web App (PWA) mit Offline-SynchronisationDas ausgelieferte Web-UI ist als PWA konzipiert. Sobald das Interface einmal im Browser geladen wurde:Service Worker speichern die gesamte App-Logik im lokalen Cache des Handys.IndexedDB / LocalStorage speichern alle eingehenden und getippten Nachrichten direkt im Handyspeicher.Asynchrone Synchronisation: Verliert das Handy den physischen Kontakt zum Knoten (weil der Nutzer weiterläuft), bleibt das Interface voll bedienbar. Kommt der Nutzer an einem anderen Knoten vorbei, synchronisiert der Browser im Hintergrund die lokal getragenen Nachrichten mit dem neuen Server (Epidemic Routing).6. Sicherheits- & Abwehr-Szenarien im ErnstfallSzenario A: Gezieltes Breitband-Jamming (Störsender)Angriff: Der Aggressor flutet das $868\text{-MHz}$-Band mit massivem Rauschen, um Rettungskräfte zu blockieren.Abwehr: SnowFoxOS-Knoten erkennen die sprunghaft ansteigende Fehlerrate und schalten automatisch (oder über das Web-UI gesteuert) auf Weg 2 (LoRa $2.4\text{ GHz}$) um. Da dieser Pfad im unüberschaubaren WLAN- und Bluetooth-Rauschen der Stadt liegt, ist er für den Feind nur unter extremem Aufwand und unter Inkaufnahme der Zerstörung eigener Funkdienste zu stören.Szenario B: Triangulation und PeilungAngriff: Feindliche Drohnen suchen nach dauerhaften Sende-Signalen, um die Position von Widerstandsnestern oder zivilen Zentralen auszumachen.Abwehr: SnowFox-Mesh verwendet extrem kurze Sende-Impulse. Da für ein P3/P5-Textpaket weniger als $150\text{ Millisekunden}$ Sendezeit benötigt werden, reicht die Zeit für feindliche Richtantennen nicht aus, um eine stabile Peilung (Triangulation) durchzuführen. Das System bleibt im "Radio-Silent"-Status, solange keine aktiven Sendungen getätigt werden.Szenario C: Kompromittierung eines KnotensAngriff: Ein Knoten wird physisch vom Feind erbeutet. Der Feind versucht, den gesamten Datenverkehr der Stadt mitzulesen.Abwehr:Öffentliche Kanäle (P2) sind ohnehin unverschlüsselt, um die Bevölkerung barrierefrei zu informieren.Behörden- und Privat-Kanäle (P3/P5) sind über das Noise Protocol Ende-zu-Ende verschlüsselt. Der Schlüssel liegt ausschließlich auf den Endgeräten der berechtigten Personen (z. B. dem Laptop des Arztes oder dem Handy des Bürgermeisters). Ein erbeuteter SnowFoxOS-Knoten enthält zu keinem Zeitpunkt die privaten Schlüssel zur Entschlüsselung dieses Datenverkehrs; er leitet die verschlüsselten Pakete lediglich blind weiter.
